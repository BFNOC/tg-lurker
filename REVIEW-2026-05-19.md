# tg-lurker 代码审查报告

**日期**: 2026-05-19  
**触发原因**: v1.5.0 引入链式合并 bug（已修复为 v1.5.1），执行全面代码审查防止类似问题  
**审查范围**: 全部核心模块（summarizer.py, database.py, scheduler.py, bot.py, dedup.py, config.py, web/）

---

## Critical — 必须修复

### C1: `_truncate_messages` 在 daily summary 路径下截断方向错误

**文件**: `summarizer.py:108-126`  
**影响**: 当消息量超过 `max_chars`（3000字符）时，daily summary 只保留当天最早的消息，丢弃最新消息

**根因**:
- `get_messages_by_date` 返回 `ORDER BY message_id DESC`
- `_truncate_messages` 的截断逻辑（`reversed(lines)` 循环）从列表末尾开始保留
- DESC 输入时，列表末尾是最旧的消息 → 截断后只剩最旧消息

**对比**: `run_incremental_summary` 使用 `get_messages_since`（ASC），截断正确保留最新消息

**复现条件**: 活跃群组单日消息超过 ~3000 字符（约 50-100 条消息）

**修复建议**: 在 `summarize_group` 中对 messages 排序为 ASC，或在 `_truncate_messages` 入口统一排序

---

### C2: `_execute_summary` 双次查询消息导致 ref_ids 与 messages 不一致

**文件**: `summarizer.py:305, 314`  
**影响**: 摘要生成和上下文窗口创建使用不同时间点的消息快照，可能导致 ref_ids 指向不存在的消息

**根因**:
```python
result = await self.summarize_group(biz_date, group_id, group_name)  # 第一次查询
# ... 中间可能有新消息写入 ...
messages = await self._db.get_messages_by_date(biz_date, group_id)   # 第二次查询
```

两次查询之间如果有新消息插入，第二次查询的 messages 集合与生成摘要时的不同。`_create_context_windows` 中的 `valid_refs` 过滤会丢弃部分 ref_ids。

**修复建议**: 让 `summarize_group` 返回使用的 messages 列表，或在 `_execute_summary` 中只查询一次

---

## High — 建议尽快修复

### H1: `cleanup_lru_contexts` 潜在无限循环

**文件**: `database.py:914-942`  
**影响**: 如果存在 context_windows 但无 context_messages 的 summary，`batch_size=0` 导致 `total` 不减少，循环不终止

**根因**:
```python
while total > max_rows:
    # ... 找到 LRU summary ...
    batch_size = count_row[0] if count_row else 0  # 可能为 0
    await self.conn.execute("DELETE FROM context_windows WHERE summary_id = ?", ...)
    total -= batch_size  # total 不变！
    deleted += batch_size
```

**修复建议**: 删除 context_windows 后，即使 batch_size=0 也应从循环中推进（例如记录已处理的 summary_id 避免重复选中）

---

### H2: `delete_messages_except_context` SQLite 参数数量限制

**文件**: `database.py:895-906`  
**影响**: 当 `keep_message_ids` 超过 999 个时，SQLite 会抛出 `too many SQL variables` 错误，导致消息清理失败

**根因**: SQLite 默认 `SQLITE_MAX_VARIABLE_NUMBER = 999`，f-string 拼接的 placeholders 可能超限

**修复建议**: 分批处理，每批不超过 900 个参数；或使用临时表存储 keep_ids

---

### H3: 密码比较存在时序攻击风险

**文件**: `web/auth.py:64`  
**影响**: 攻击者可通过响应时间差异逐字符猜测密码

**根因**: `password == request.app.state.config.web_password` 使用短路比较

**修复建议**: 使用 `secrets.compare_digest(password, config.web_password)`

---

## Medium — 建议修复

### M1: `_admin_cache` 无界增长

**文件**: `bot.py:30`  
**影响**: 长时间运行后，每个 (chat_id, sender_id) 对永久占用内存

**修复建议**: 添加 LRU 淘汰或定期清理过期条目（当前 TTL 5 分钟但从不删除过期项）

---

### M2: `_get_client` 缓存客户端从未使用

**文件**: `summarizer.py:43-55, 88-106`  
**影响**: 每次 LLM 调用都创建新的 `AsyncOpenAI` 客户端并关闭，无法利用连接池，增加延迟

**根因**: `_call_llm` 方法每次创建新 client（支持热更新配置），但 `_get_client` 方法成为死代码

**修复建议**: 删除 `_get_client` 死代码，或在配置未变时复用 client

---

### M3: bot.py 中 dedup hash 移除逻辑脆弱

**文件**: `bot.py:235`  
**影响**: 使用 `__import__("dedup")` 内联导入且直接操作私有属性

```python
self._dedup._hashes.discard(__import__("dedup").text_hash(text))
```

**修复建议**: 在 `DedupEngine` 上添加 `remove_hash(text)` 公开方法

---

### M4: scheduler 中 custom group 消息清理与并发摘要的竞态

**文件**: `scheduler.py:167-175`  
**影响**: `_clear_custom_group_messages` 在全局摘要完成后清理自定义频率群组的消息，但该群组可能正在执行自己的摘要任务

**修复建议**: 清理前检查该群组是否有正在运行的摘要任务，或使用 per-group 锁

---

## Low — 可选改进

### L1: `_truncate_messages` 使用 `insert(0, ...)` 性能差

**文件**: `summarizer.py:110-114`  
**影响**: `list.insert(0, x)` 是 O(n) 操作，对大消息列表有性能影响

**修复建议**: 使用 `collections.deque` 或先 append 再 reverse

---

### L2: 缺少 `_group_nearby_refs` 的单元测试

**文件**: 无测试文件  
**影响**: v1.5.0 的链式合并 bug 本可通过简单单元测试发现

**修复建议**: 添加测试覆盖：
- 链式 refs 不应全合并（本次 bug 场景）
- 距离小于阈值的 refs 应合并
- 空 refs / 重复 refs 边界情况

---

### L3: `get_messages_by_date` 排序方向不一致

**文件**: `database.py:265` vs `database.py:298`  
**影响**: `get_messages_by_date` 返回 DESC，`get_messages_since` 返回 ASC，调用方需要了解这个差异

**修复建议**: 统一为 ASC，或在方法名/文档中明确排序方向
